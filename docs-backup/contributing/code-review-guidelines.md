# 程式碼審查指南

> PR 審查的標準與流程

---

## 🎯 審查重點

### 1. 功能正確性
- [ ] 功能符合需求
- [ ] 邊界情況已處理
- [ ] 錯誤處理適當

### 2. 程式碼品質
- [ ] 遵循編碼標準
- [ ] 命名清晰有意義
- [ ] 沒有重複代碼
- [ ] 複雜度合理

### 3. 測試
- [ ] 有對應的測試
- [ ] 測試覆蓋關鍵路徑
- [ ] 測試通過

### 4. 安全性
- [ ] 無 SQL 注入風險
- [ ] 無 XSS 風險
- [ ] 敏感資料處理正確
- [ ] RLS 政策驗證

### 5. 效能
- [ ] 無明顯效能問題
- [ ] 資料庫查詢優化
- [ ] 避免不必要的渲染

### 6. 文件
- [ ] 程式碼註解清楚
- [ ] README 更新（如需要）
- [ ] API 文件更新（如需要）

---

## 💬 審查回饋

### 好的回饋
- 具體指出問題
- 提供改善建議
- 尊重對方

### 範例

❌ 不好：
> 這段代碼很糟糕

✅ 好：
> 建議將這個函數拆分為更小的單元，以提高可讀性和可測試性

---

## 🔄 審查流程

1. 收到 PR 通知
2. 閱讀 PR 描述
3. 檢視變更的檔案
4. 執行本地測試（如需要）
5. 提供回饋
6. 批准或請求修改
